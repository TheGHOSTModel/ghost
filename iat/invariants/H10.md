# H10 — Enforcement attributable

> Enforcement actions must be attributable and reviewable

## Why this matters

Every enforcement action must carry actor, reason, evidence reference, and timestamp. Untracked bans and phantom moderation collapse appeal and audit.

## Telemetry

**Event:** `enforcement_decision`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `subject_id` | string | yes | identifier |
| `action` | string | yes | value |
| `actor` | string | yes | value |
| `reason_code` | string | yes | value |
| `evidence_ref` | string | yes | value |
| `ts` | string | yes | ISO-8601 timestamp |

**Example payload:**

```json
{
  "event": "enforcement_decision",
  "subject_id": "<subject_id>",
  "action": "<action>",
  "actor": "<actor>",
  "reason_code": "<reason_code>",
  "evidence_ref": "<evidence_ref>",
  "ts": "2026-05-27T12:34:56Z"
}
```

## Detection signal

```
Action without actor, reason, or evidence reference
```

## Threats this catches

- Untracked ban
- Phantom moderation
- Unaccountable action

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Every enforcement decision carries actor, action, reason code, evidence reference, and timestamp in a single event. |
| 🟡 **Latent** | Enforcement happens but emitted events are missing one or more of the required attribution fields. |
| 🔵 **Collapsed** | Covered by S09 — moderation actions go through the same community-action log. |
| ⚪ **Inactive (N/A)** | No enforcement actions are taken — the application has no moderation surface. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
