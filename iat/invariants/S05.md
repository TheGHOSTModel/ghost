# S05 — New identities lack instant power

> New identities must not gain instant influence

## Why this matters

High-influence privileges must be age-gated. Throwaway accounts gaining instant power is the throwaway-abuse pattern.

## Telemetry

**Event:** `privilege_check`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `player_id` | string | yes | identifier |
| `privilege` | string | yes | value |
| `account_age_days` | string | yes | value |
| `decision` | string | yes | decision/outcome enum |

**Example payload:**

```json
{
  "event": "privilege_check",
  "player_id": "<player_id>",
  "privilege": "<privilege>",
  "account_age_days": "<account_age_days>",
  "decision": "allow"
}
```

## Detection signal

```
High-influence privilege granted to account_age < threshold
```

## Threats this catches

- Throwaway-account abuse
- Fast-track impersonation
- Sleeper exploit

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Privilege checks evaluate account_age_days against thresholds and emit the decision per privilege. |
| 🟡 **Latent** | Account age is tracked but not consulted in privilege evaluation. |
| 🔵 **Collapsed** | Covered by S07 — privilege evaluation is one branch of the role/hierarchy check. |
| ⚪ **Inactive (N/A)** | No privileges that depend on account standing — all accounts have identical permissions. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
