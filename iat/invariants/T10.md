# T10 — Economy-wide anomalies detectable

> Economy-wide anomalies must be detectable

## Why this matters

Economy-wide health metrics (inflation, drain, scarcity) must be sampled with baseline comparison. Anomaly scores above threshold catch macro shocks before users do.

## Telemetry

**Event:** `economy_health_sample`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `metric` | string | yes | value |
| `value` | number | yes | numeric value |
| `baseline` | number | yes | numeric value |
| `anomaly_score` | number | yes | numeric value |
| `ts` | string | yes | ISO-8601 timestamp |

**Example payload:**

```json
{
  "event": "economy_health_sample",
  "metric": "<metric>",
  "value": 0.87,
  "baseline": 0.87,
  "anomaly_score": 0.87,
  "ts": "2026-05-27T12:34:56Z"
}
```

## Detection signal

```
anomaly_score above threshold for any economy_health metric
```

## Threats this catches

- Hyperinflation
- Sudden scarcity
- Currency drain
- Market shock

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Economy-wide metrics are sampled per window with baseline comparison and anomaly scoring. |
| 🟡 **Latent** | Economy metrics are tracked but baselines or anomaly scoring are missing. |
| 🔵 **Collapsed** | Covered by O10 — economy metrics are emitted as part of the service-health SLI stream. |
| ⚪ **Inactive (N/A)** | No economy to monitor — see T01. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
