# O01 — External inputs validated

> All external inputs must be validated

## Why this matters

All external inputs (HTTP, RPC, message bus) must pass a declared validator before reaching business logic. A spike in rejections is intel; the absence of a validator is a vulnerability.

## Telemetry

**Event:** `input_validation_result`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `endpoint` | string | yes | value |
| `validator_id` | string | yes | identifier |
| `decision` | string | yes | decision/outcome enum |
| `payload_size` | integer | yes | numeric value |
| `ts` | string | yes | ISO-8601 timestamp |

**Example payload:**

```json
{
  "event": "input_validation_result",
  "endpoint": "<endpoint>",
  "validator_id": "<validator_id>",
  "decision": "allow",
  "payload_size": 42,
  "ts": "2026-05-27T12:34:56Z"
}
```

## Detection signal

```
decision == 'rejected' rate spike OR validator absent for endpoint
```

## Threats this catches

- Malformed payload exploit
- Schema-fuzzing
- Injection attempt

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Every external endpoint declares a validator and validation outcomes are emitted with decision and payload size. |
| 🟡 **Latent** | Validators exist but outcomes are not emitted as structured events. |
| 🔵 **Collapsed** | Covered by O02 (parse outcome) — validation and parsing share the same telemetry pipe. |
| ⚪ **Inactive (N/A)** | No external inputs accepted — the system is fully internal or read-only with no inbound writes. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
