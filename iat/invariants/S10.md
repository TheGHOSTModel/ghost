# S10 — Social enforcement visible

> Social enforcement must be visible

## Why this matters

Enforcement that the policy requires to be visible (e.g. public warnings) must emit a public_visible=true with audit_ref. The signal catches shadow bans where the policy expected transparency.

## Telemetry

**Event:** `enforcement_visible`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `subject_id` | string | yes | identifier |
| `action_class` | string | yes | value |
| `public_visible` | boolean | yes | true/false |
| `audit_ref` | string | yes | value |

**Example payload:**

```json
{
  "event": "enforcement_visible",
  "subject_id": "<subject_id>",
  "action_class": "<action_class>",
  "public_visible": true,
  "audit_ref": "<audit_ref>"
}
```

## Detection signal

```
Enforcement action with public_visible == false where policy requires visibility
```

## Threats this catches

- Shadow ban without audit
- Invisible enforcement
- Policy drift

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Enforcement actions emit a public_visible field and an audit_ref, and policy-required visibility is checked. |
| 🟡 **Latent** | Enforcement is emitted but visibility status is not part of the event. |
| 🔵 **Collapsed** | Covered by H10 — enforcement attribution already carries visibility metadata. |
| ⚪ **Inactive (N/A)** | No social enforcement is taken — see H10. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
