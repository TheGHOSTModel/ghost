# G09 — Client state not blindly trusted

> Client-reported state must never be blindly trusted

## Why this matters

Any value the client asserts about its own state must be reconciled against server truth. Acceptance of unreconciled client values is the root cause of most memory-edit and state-forgery attacks.

## Telemetry

**Event:** `client_claim`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `player_id` | string | yes | identifier |
| `claim_type` | string | yes | value |
| `claim_value` | string | yes | value |
| `reconciliation_outcome` | string | yes | decision/outcome enum |

**Example payload:**

```json
{
  "event": "client_claim",
  "player_id": "<player_id>",
  "claim_type": "<claim_type>",
  "claim_value": "<claim_value>",
  "reconciliation_outcome": "<reconciliation_outcome>"
}
```

## Detection signal

```
Server reconciles to a different value OR rejects claim
```

## Threats this catches

- Client-side cheating
- Memory editing
- State forgery

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Every client-asserted state value is reconciled against server truth before being trusted, and reconciliation outcomes are logged. |
| 🟡 **Latent** | Client claims are accepted in the live path but a parallel reconciliation runs offline. |
| 🔵 **Collapsed** | Covered by G01 — the server simply does not accept client-claimed state. |
| ⚪ **Inactive (N/A)** | Clients are never asked to report their own state — the server holds and emits everything. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
