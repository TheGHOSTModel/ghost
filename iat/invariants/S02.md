# S02 — Reputation hard to game

> Trust and reputation must be difficult to game

## Why this matters

Reputation deltas must carry an attestable source_event with a non-self-referential provenance chain. Otherwise rep farming and sockpuppet endorsement are trivial.

## Telemetry

**Event:** `reputation_delta`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `player_id` | string | yes | identifier |
| `delta` | number | yes | numeric value |
| `source_event` | string | yes | value |
| `provenance_chain` | string | yes | value |

**Example payload:**

```json
{
  "event": "reputation_delta",
  "player_id": "<player_id>",
  "delta": 0.87,
  "source_event": "<source_event>",
  "provenance_chain": "<provenance_chain>"
}
```

## Detection signal

```
Reputation gain without valid source_event OR self-referential chain
```

## Threats this catches

- Rep farming
- Fake review
- Sockpuppet endorsement

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Reputation deltas emit with a source_event reference and a provenance chain that can be audited for self-reference. |
| 🟡 **Latent** | Reputation changes are emitted but provenance is not captured. |
| 🔵 **Collapsed** | Covered by S06 — every reputation event already carries source_action_id and actor_id. |
| ⚪ **Inactive (N/A)** | No reputation or trust system exists in the application. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
