# G10 — Debug paths isolated from prod

> Non-production or debug paths must not affect live activity

## Why this matters

Debug, QA, and admin code paths must be unreachable from live sessions. A debug-tagged path invoked from production telemetry is the canonical privileged-access leak.

## Telemetry

**Event:** `code_path_invocation`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `path_id` | string | yes | identifier |
| `env_tag` | string | yes | value |
| `caller` | string | yes | value |
| `ts` | string | yes | ISO-8601 timestamp |

**Example payload:**

```json
{
  "event": "code_path_invocation",
  "path_id": "<path_id>",
  "env_tag": "<env_tag>",
  "caller": "<caller>",
  "ts": "2026-05-27T12:34:56Z"
}
```

## Detection signal

```
env_tag == 'debug' invoked from production session
```

## Threats this catches

- Debug-flag abuse
- Cheat-menu access
- QA tooling exposure

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Debug, QA, and admin code paths are tagged and any invocation from a production session emits an alert. |
| 🟡 **Latent** | Debug paths are tagged but invocations are not monitored. |
| 🔵 **Collapsed** | Covered by O01 (input validation) — debug endpoints are unreachable from the production gateway, validated at the edge. |
| ⚪ **Inactive (N/A)** | No debug or non-production code is shipped in the production binary. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
