# T03 — Transfers atomic and consistent

> Transfers must be atomic and consistent

## Why this matters

Two-phase or transactional semantics on every transfer. A debit without a matching credit (or vice versa) without rollback is lost or phantom value.

## Telemetry

**Event:** `transfer_outcome`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `transfer_id` | string | yes | identifier |
| `debit_outcome` | string | yes | decision/outcome enum |
| `credit_outcome` | string | yes | decision/outcome enum |
| `atomic_flag` | boolean | yes | true/false |

**Example payload:**

```json
{
  "event": "transfer_outcome",
  "transfer_id": "<transfer_id>",
  "debit_outcome": "<debit_outcome>",
  "credit_outcome": "<credit_outcome>",
  "atomic_flag": true
}
```

## Detection signal

```
Debit succeeded but credit failed (or vice versa) without rollback
```

## Threats this catches

- Partial transfer
- Lost item
- Phantom credit

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Transfers emit both debit and credit outcomes with an atomic flag, and partial-success cases alert. |
| 🟡 **Latent** | Transfers are logged but debit and credit are not paired in the event stream. |
| 🔵 **Collapsed** | Covered by T01 — atomicity is enforced at the ledger sequence level. |
| ⚪ **Inactive (N/A)** | No multi-account transfers exist — see T01. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
