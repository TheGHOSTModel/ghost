# O02 — Tolerate malformed input

> Systems must tolerate malformed or hostile input

## Why this matters

Parsers must fail closed and recover. An unhandled parse exception that crashes the worker is a denial-of-service primitive.

## Telemetry

**Event:** `input_parse_outcome`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `endpoint` | string | yes | value |
| `parse_result` | string | yes | value |
| `exception_type` | string | yes | value |
| `recovery_action` | string | yes | value |

**Example payload:**

```json
{
  "event": "input_parse_outcome",
  "endpoint": "<endpoint>",
  "parse_result": "<parse_result>",
  "exception_type": "<exception_type>",
  "recovery_action": "<recovery_action>"
}
```

## Detection signal

```
Unhandled exception leading to crash OR cascading parse failures
```

## Threats this catches

- Parser crash exploit
- Stack-overflow input
- Malformed JSON DOS

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Parse outcomes are emitted with result, exception class if any, and recovery action, and unhandled exceptions trigger alerts. |
| 🟡 **Latent** | Parse failures are caught but not emitted with structured fields. |
| 🔵 **Collapsed** | Covered by O08 (error event) — parse failures are subsumed into the general error stream. |
| ⚪ **Inactive (N/A)** | No parsing of external input — input is structured and pre-validated by a gateway. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
