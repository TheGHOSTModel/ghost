# O07 — Operational limits enforced

> Operational limits must be enforced

## Why this matters

Quotas and operational limits must exist for every unbounded action class. The absence of a limit is itself a signal.

## Telemetry

**Event:** `limit_enforcement`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `subject` | string | yes | value |
| `limit_type` | string | yes | value |
| `current` | number | yes | numeric value |
| `limit` | number | yes | numeric value |
| `decision` | string | yes | decision/outcome enum |

**Example payload:**

```json
{
  "event": "limit_enforcement",
  "subject": "<subject>",
  "limit_type": "<limit_type>",
  "current": 0.87,
  "limit": 0.87,
  "decision": "allow"
}
```

## Detection signal

```
decision == 'denied' rate climbing OR limit absent for action type
```

## Threats this catches

- Quota exhaustion
- Unbounded operation
- Limit-config drift

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Every operational action class has a declared limit, enforcement decisions are emitted, and rising denial rates alert. |
| 🟡 **Latent** | Limits exist for some action classes but not all, or decisions are not emitted as events. |
| 🔵 **Collapsed** | Covered by O03 — operational limits and rate limits share the same enforcement telemetry. |
| ⚪ **Inactive (N/A)** | The application has no unbounded actions to limit. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
