# H09 — Safety controls resist evasion

> Safety controls must resist evasion

## Why this matters

Safety controls must hold under adversarial pressure. The same input bypassing on retry, or an evasion score climbing across attempts, indicates the control is being mapped.

## Telemetry

**Event:** `control_evaluation`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `control_id` | string | yes | identifier |
| `input_signature` | string | yes | value |
| `decision` | string | yes | decision/outcome enum |
| `evasion_score` | number | yes | numeric value |

**Example payload:**

```json
{
  "event": "control_evaluation",
  "control_id": "<control_id>",
  "input_signature": "<input_signature>",
  "decision": "allow",
  "evasion_score": 0.87
}
```

## Detection signal

```
Same input bypassing control on retry OR evasion_score climbing
```

## Threats this catches

- Filter bypass
- Unicode obfuscation
- Rate-limit dodging

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Safety control evaluations emit input signature, decision, and an evasion score that climbs on repeated near-miss attempts. |
| 🟡 **Latent** | Controls evaluate but evasion scoring is not implemented. |
| 🔵 **Collapsed** | Covered by O01 (input validation) — controls are evaluated at the input boundary and reuse that telemetry. |
| ⚪ **Inactive (N/A)** | No content or behavior controls are deployed — the application has no safety surface to evade. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
