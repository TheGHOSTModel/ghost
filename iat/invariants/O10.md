# O10 — Degradation detectable

> Degraded operation must be detectable

## Why this matters

SLI/SLO-anchored health per service. Degradation is not binary; status crossing threshold sustained is the signal.

## Telemetry

**Event:** `service_health`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `service` | string | yes | value |
| `sli_name` | string | yes | value |
| `value` | number | yes | numeric value |
| `threshold` | string | yes | value |
| `status` | string | yes | decision/outcome enum |

**Example payload:**

```json
{
  "event": "service_health",
  "service": "<service>",
  "sli_name": "<sli_name>",
  "value": 0.87,
  "threshold": "<threshold>",
  "status": "<status>"
}
```

## Detection signal

```
status == 'degraded' OR value crossing threshold sustained
```

## Threats this catches

- Silent degradation
- Performance cliff
- Latency creep

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | SLIs per service are sampled at a regular interval with threshold and status fields emitted in every sample. |
| 🟡 **Latent** | SLIs are sampled but status field or threshold comparison is missing. |
| 🔵 **Collapsed** | Covered by O08 — error rates alone serve as the SLI and no separate health emission is needed. |
| ⚪ **Inactive (N/A)** | The system has no production deployment to monitor — pre-launch or research-only. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
