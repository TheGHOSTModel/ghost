# G08 — Actions not replayable

> Actions must not be replayed to gain additional benefit

## Why this matters

Each action must be uniquely identifiable and accepted at most once per session. Replays of the same nonce or action_id indicate token reuse or duplicate-action exploits.

## Telemetry

**Event:** `action_submission`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `action_id` | string | yes | identifier |
| `nonce` | string | yes | value |
| `player_id` | string | yes | identifier |
| `session_id` | string | yes | identifier |
| `ts` | string | yes | ISO-8601 timestamp |

**Example payload:**

```json
{
  "event": "action_submission",
  "action_id": "<action_id>",
  "nonce": "<nonce>",
  "player_id": "<player_id>",
  "session_id": "<session_id>",
  "ts": "2026-05-27T12:34:56Z"
}
```

## Detection signal

```
Repeated nonce or action_id within active session
```

## Threats this catches

- Replay attack
- Duplicate-action exploit
- Token reuse

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Every action submission carries a nonce or action_id, and repeated values within an active session trigger rejection and alert. |
| 🟡 **Latent** | Nonces are required by the protocol but not checked for replay in the live path. |
| 🔵 **Collapsed** | Covered by G01 — server authority rejects replayed actions implicitly and emits the same telemetry. |
| ⚪ **Inactive (N/A)** | Actions are inherently idempotent and replay produces no additional effect. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
