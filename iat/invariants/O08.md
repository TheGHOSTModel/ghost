# O08 — Failures emit diagnostics

> Failures must produce diagnostic telemetry

## Why this matters

Every error must carry service, class, correlation_id, and stack ref. Generic "unknown" error_class destroys post-incident traceability.

## Telemetry

**Event:** `error_event`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `service` | string | yes | value |
| `error_class` | string | yes | value |
| `correlation_id` | string | yes | identifier |
| `stack_ref` | string | yes | value |
| `ts` | string | yes | ISO-8601 timestamp |

**Example payload:**

```json
{
  "event": "error_event",
  "service": "<service>",
  "error_class": "<error_class>",
  "correlation_id": "<correlation_id>",
  "stack_ref": "<stack_ref>",
  "ts": "2026-05-27T12:34:56Z"
}
```

## Detection signal

```
Errors without correlation_id OR generic 'unknown' error_class
```

## Threats this catches

- Silent failure
- Black-box error
- Missing trace

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Errors are emitted with service, error class, correlation_id, and stack reference, and generic error_class is treated as a bug. |
| 🟡 **Latent** | Errors are emitted but missing one or more required fields, or correlation_id is not propagated. |
| 🔵 **Collapsed** | Covered by O09 — errors produce recovery events that already carry the required diagnostic fields. |
| ⚪ **Inactive (N/A)** | No error handling required — the application has no failure modes that need diagnosis. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
