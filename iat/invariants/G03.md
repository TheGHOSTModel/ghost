# G03 — Consistent rule enforcement

> Rules must be enforced consistently across users

## Why this matters

Rule evaluation must be deterministic and identical for all users given equivalent inputs. Divergent decisions for the same rule_id reveal config drift, experiment leakage, or privileged code paths.

## Telemetry

**Event:** `rule_evaluation`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `player_id` | string | yes | identifier |
| `rule_id` | string | yes | identifier |
| `decision` | string | yes | decision/outcome enum |
| `ts` | string | yes | ISO-8601 timestamp |

**Example payload:**

```json
{
  "event": "rule_evaluation",
  "player_id": "<player_id>",
  "rule_id": "<rule_id>",
  "decision": "allow",
  "ts": "2026-05-27T12:34:56Z"
}
```

## Detection signal

```
Same rule_id producing divergent decisions for equivalent inputs
```

## Threats this catches

- Rule bypass
- Privileged-account drift
- A/B-config leak

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Rule evaluations are logged with rule_id and decision, and divergent decisions for equivalent inputs trigger investigation. |
| 🟡 **Latent** | Rule decisions are logged but not compared across users or experiments to detect drift. |
| 🔵 **Collapsed** | A central policy engine emits decisions that subsume this — e.g. all rule paths funnel through a single audit event already covered elsewhere. |
| ⚪ **Inactive (N/A)** | No conditional rules to evaluate — the application has only unconditional flows. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
