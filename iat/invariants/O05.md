# O05 — Resource exhaustion detectable

> Resource exhaustion must be detectable

## Why this matters

Resource utilization (memory, FDs, connections, threads) must be sampled and bounded. Growth rate is more diagnostic than instantaneous level for leak detection.

## Telemetry

**Event:** `resource_utilization`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `resource` | string | yes | value |
| `current` | number | yes | numeric value |
| `limit` | number | yes | numeric value |
| `growth_rate` | number | yes | numeric value |
| `ts` | string | yes | ISO-8601 timestamp |

**Example payload:**

```json
{
  "event": "resource_utilization",
  "resource": "<resource>",
  "current": 0.87,
  "limit": 0.87,
  "growth_rate": 0.87,
  "ts": "2026-05-27T12:34:56Z"
}
```

## Detection signal

```
growth_rate > threshold OR current/limit > 0.85 sustained
```

## Threats this catches

- Memory leak
- File-descriptor exhaustion
- Connection-pool starvation

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Resource utilization is sampled per resource and growth-rate alerts fire before exhaustion. |
| 🟡 **Latent** | Utilization is sampled but growth rate is not derived or thresholded. |
| 🔵 **Collapsed** | Covered by O10 — resource SLIs are emitted as part of service health. |
| ⚪ **Inactive (N/A)** | The platform manages resources for the application (fully serverless) and resource exhaustion is the provider’s responsibility. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
