# G04 — State synchronization

> Shared state must remain synchronized across participants

## Why this matters

All participants in a session must converge on the same authoritative state. Hash divergence between client and server indicates desync, injection, or phantom-action exploits.

## Telemetry

**Event:** `state_sync_event`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `session_id` | string | yes | identifier |
| `server_state_hash` | string | yes | value |
| `client_state_hash` | string | yes | value |
| `ts` | string | yes | ISO-8601 timestamp |

**Example payload:**

```json
{
  "event": "state_sync_event",
  "session_id": "<session_id>",
  "server_state_hash": "<server_state_hash>",
  "client_state_hash": "<client_state_hash>",
  "ts": "2026-05-27T12:34:56Z"
}
```

## Detection signal

```
client_state_hash != server_state_hash for any participant
```

## Threats this catches

- Desync exploitation
- State injection
- Phantom moves

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Authoritative state is hashed and compared against client-reported state on every sync, with mismatches surfaced. |
| 🟡 **Latent** | State hashing exists in the server but is not compared against clients, or the comparison happens but is not logged. |
| 🔵 **Collapsed** | Covered by G01 (server authority) — every state change is server-validated and clients have no independent state. |
| ⚪ **Inactive (N/A)** | Each user operates on independent state with no shared world — single-player or strictly per-user views. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
