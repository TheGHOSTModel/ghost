# H07 — Users can report harm

> Users must be able to report harmful behavior

## Why this matters

Users must have a reachable, structured reporting channel that emits to a real queue. Reporting UI that drops to /dev/null is a known anti-pattern; the report_submitted event proves the path is alive.

## Telemetry

**Event:** `report_submitted`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `reporter_id` | string | yes | identifier |
| `subject_id` | string | yes | identifier |
| `category` | string | yes | value |
| `evidence_ref` | string | yes | value |
| `ts` | string | yes | ISO-8601 timestamp |

**Example payload:**

```json
{
  "event": "report_submitted",
  "reporter_id": "<reporter_id>",
  "subject_id": "<subject_id>",
  "category": "<category>",
  "evidence_ref": "<evidence_ref>",
  "ts": "2026-05-27T12:34:56Z"
}
```

## Detection signal

```
Reporting UI exists, emits to a queue with non-zero processing rate
```

## Threats this catches

- Reporting UI absent
- Silent drop
- No category for harm type

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | A reporting UI exists, submissions emit structured events, and the downstream queue has non-zero processing. |
| 🟡 **Latent** | Reporting UI exists but submissions do not emit telemetry, or the queue is unmonitored. |
| 🔵 **Collapsed** | Covered by H08 — every submission produces a state-change event already. |
| ⚪ **Inactive (N/A)** | No user-to-user interaction or user-visible content for users to report against. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
