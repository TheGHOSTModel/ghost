# G06 — Validated progression only

> Persistent progression must advance only through validated actions

## Why this matters

Every progression delta must trace back to a validated action lineage. A new state without a chain to an authoritative cause is unearned progress.

## Telemetry

**Event:** `progression_change`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `player_id` | string | yes | identifier |
| `prior_state` | string | yes | value |
| `new_state` | string | yes | value |
| `derived_from_action_id` | string | yes | identifier |

**Example payload:**

```json
{
  "event": "progression_change",
  "player_id": "<player_id>",
  "prior_state": "<prior_state>",
  "new_state": "<new_state>",
  "derived_from_action_id": "<derived_from_action_id>"
}
```

## Detection signal

```
new_state without matching validated action_id in lineage
```

## Threats this catches

- Save-state injection
- Progression skipping
- Achievement forgery

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Every persistent progression mutation is logged with a reference to the validated action that caused it, and orphan mutations alert. |
| 🟡 **Latent** | Progression changes are logged but without lineage to a triggering action. |
| 🔵 **Collapsed** | Covered by G01 — every progression delta is itself a server-validated action with no separate lineage needed. |
| ⚪ **Inactive (N/A)** | No persistent progression — state resets between sessions or there is no concept of advancement. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
