# S01 — Identity not impersonable

> User identity must not be impersonable

## Why this matters

Identity creation and rename must compute similarity against existing handles and emit a decision. Look-alike attacks are the primary impersonation vector at scale.

## Telemetry

**Event:** `identity_create_or_rename`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `player_id` | string | yes | identifier |
| `handle` | string | yes | value |
| `similarity_to_existing` | number | yes | numeric value |
| `decision` | string | yes | decision/outcome enum |

**Example payload:**

```json
{
  "event": "identity_create_or_rename",
  "player_id": "<player_id>",
  "handle": "<handle>",
  "similarity_to_existing": 0.87,
  "decision": "allow"
}
```

## Detection signal

```
similarity score above threshold OR no guard evaluated
```

## Threats this catches

- Look-alike username
- Handle squat
- Impersonation account

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Identity creation and rename requests evaluate similarity to existing handles and emit the decision with score. |
| 🟡 **Latent** | Similarity checks run but decisions are not emitted as events. |
| 🔵 **Collapsed** | Covered by H06 — identity changes go through the same validator and emit there. |
| ⚪ **Inactive (N/A)** | No user-controlled identity — handles are system-assigned and unique by construction. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
