# G02 — Inputs bounded to human limits

> User inputs must not exceed plausible human rates

## Why this matters

Human users are bounded by reflex and motor limits. Sustained input rates above the human p99 envelope indicate automation, not skill. The bound is per-action-type, per-time-window.

## Telemetry

**Event:** `input_rate_sample`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `player_id` | string | yes | identifier |
| `action_type` | string | yes | value |
| `count_per_window` | integer | yes | numeric value |
| `window_ms` | integer | yes | numeric value |

**Example payload:**

```json
{
  "event": "input_rate_sample",
  "player_id": "<player_id>",
  "action_type": "<action_type>",
  "count_per_window": 42,
  "window_ms": 42
}
```

## Detection signal

```
count_per_window > human_p99_threshold sustained over N windows
```

## Threats this catches

- Input automation
- Macro use
- Scripted clicking

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Per-user, per-action input rates are sampled and compared to a human-plausible upper bound, with alerts on sustained breaches. |
| 🟡 **Latent** | Rate sampling is collected but not compared to a threshold, or a threshold exists but does not page. |
| 🔵 **Collapsed** | Folded into a broader anti-automation signal (e.g. CAPTCHA gates or G05 timing detection) that already catches the same threat class. |
| ⚪ **Inactive (N/A)** | No user input loop with competitive stakes — for example, a read-only dashboard or a system with no per-user actions. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
