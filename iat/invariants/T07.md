# T07 — Collusion leaves signals

> Collusive behavior must leave observable signals

## Why this matters

Cross-account outcome correlation in competitive or transactional contexts. Skewed outcomes without organic explanation across N events is collusion or boosting.

## Telemetry

**Event:** `match_outcome_correlation`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `account_pair` | array | yes | list of identifiers |
| `outcome_skew` | number | yes | numeric value |
| `frequency` | integer | yes | numeric value |
| `window` | integer | yes | numeric value |

**Example payload:**

```json
{
  "event": "match_outcome_correlation",
  "account_pair": [
    "<id_1>",
    "<id_2>"
  ],
  "outcome_skew": 0.87,
  "frequency": 42,
  "window": 42
}
```

## Detection signal

```
outcome_skew biased without organic explanation across N matches
```

## Threats this catches

- Match throwing
- Win-trading
- Boosting collusion

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Outcome correlation across account pairs is sampled and scored, and skewed outcomes without organic explanation alert. |
| 🟡 **Latent** | Outcome data is collected but cross-account correlation is not yet computed. |
| 🔵 **Collapsed** | Covered by S04 — coordination signals already detect the collusive accounts. |
| ⚪ **Inactive (N/A)** | No competitive outcomes or transactional pairings exist between accounts. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
