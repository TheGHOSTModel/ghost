# H08 — Reports handled observably

> Reports must result in observable handling

## Why this matters

Every report must traverse a state machine to a terminal state. Reports stuck past SLA, or queues with no terminal transitions, are the "black hole" failure mode.

## Telemetry

**Event:** `report_state_change`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `report_id` | string | yes | identifier |
| `prior_state` | string | yes | value |
| `new_state` | string | yes | value |
| `actor` | string | yes | value |
| `ts` | string | yes | ISO-8601 timestamp |

**Example payload:**

```json
{
  "event": "report_state_change",
  "report_id": "<report_id>",
  "prior_state": "<prior_state>",
  "new_state": "<new_state>",
  "actor": "<actor>",
  "ts": "2026-05-27T12:34:56Z"
}
```

## Detection signal

```
Report stuck in queue past SLA OR terminal state never reached
```

## Threats this catches

- Report black hole
- Untracked closure
- Stale queue

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Reports traverse an observable state machine and emit events on every transition, with SLA monitoring on terminal-state reach. |
| 🟡 **Latent** | Reports change state but transitions are not logged as discrete events. |
| 🔵 **Collapsed** | Covered by H10 — every report resolves into an enforcement decision emitted there. |
| ⚪ **Inactive (N/A)** | No reporting flow exists — see H07. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
