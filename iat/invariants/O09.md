# O09 — Recovery is visible

> Recovery behavior must be visible

## Why this matters

Recovery actions (failover, retry, restart) must emit trigger and outcome. Recovery without outcome telemetry hides stuck states.

## Telemetry

**Event:** `recovery_event`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `service` | string | yes | value |
| `trigger` | string | yes | value |
| `recovery_action` | string | yes | value |
| `outcome` | string | yes | value |
| `duration_ms` | integer | yes | numeric value |

**Example payload:**

```json
{
  "event": "recovery_event",
  "service": "<service>",
  "trigger": "<trigger>",
  "recovery_action": "<recovery_action>",
  "outcome": "<outcome>",
  "duration_ms": 42
}
```

## Detection signal

```
Recovery triggered without outcome emission OR repeated triggers
```

## Threats this catches

- Stuck recovery
- Recovery loop
- Silent fail-over

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Recovery triggers and outcomes are emitted as paired events, and recovery without outcome alerts. |
| 🟡 **Latent** | Recovery actions happen but outcomes are not emitted alongside triggers. |
| 🔵 **Collapsed** | Covered by O10 — recovery is reflected in service-health SLI transitions. |
| ⚪ **Inactive (N/A)** | No recovery mechanisms — the application has no automated retry, failover, or restart logic. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
