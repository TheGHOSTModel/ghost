# S06 — Reputation changes attributable

> Reputation changes must be attributable

## Why this matters

Every reputation change must have a source_action_id and a matching actor_id. Off-platform manipulation is invisible without this anchor.

## Telemetry

**Event:** `reputation_event`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `subject_id` | string | yes | identifier |
| `prior` | number | yes | numeric value |
| `new` | number | yes | numeric value |
| `source_action_id` | string | yes | identifier |
| `actor_id` | string | yes | identifier |

**Example payload:**

```json
{
  "event": "reputation_event",
  "subject_id": "<subject_id>",
  "prior": 0.87,
  "new": 0.87,
  "source_action_id": "<source_action_id>",
  "actor_id": "<actor_id>"
}
```

## Detection signal

```
Change without source_action_id OR actor_id mismatched to source
```

## Threats this catches

- Phantom rep change
- Off-platform manipulation
- Untracked endorsement

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Every reputation change emits subject, prior, new, source_action_id, and actor_id in one event. |
| 🟡 **Latent** | Reputation events are emitted but one or more provenance fields are missing. |
| 🔵 **Collapsed** | Covered by S02 — reputation deltas carry provenance chains that subsume this. |
| ⚪ **Inactive (N/A)** | No reputation system exists. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
