# T09 — Rollbacks auditable

> Rollbacks or reversals must be auditable

## Why this matters

Every rollback or reversal must reference the original_action_id and the actor performing the reversal. Untracked restores are the audit-evasion pattern.

## Telemetry

**Event:** `reversal_event`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `reversal_id` | string | yes | identifier |
| `original_action_id` | string | yes | identifier |
| `reason` | string | yes | value |
| `actor` | string | yes | value |
| `ts` | string | yes | ISO-8601 timestamp |

**Example payload:**

```json
{
  "event": "reversal_event",
  "reversal_id": "<reversal_id>",
  "original_action_id": "<original_action_id>",
  "reason": "<reason>",
  "actor": "<actor>",
  "ts": "2026-05-27T12:34:56Z"
}
```

## Detection signal

```
Reversal without original_action_id reference OR actor null
```

## Threats this catches

- Untracked rollback
- Silent reversal
- Audit-evading restore

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Every rollback or reversal emits original_action_id, reason, actor, and timestamp. |
| 🟡 **Latent** | Reversals happen but original_action_id is not consistently emitted. |
| 🔵 **Collapsed** | Covered by T05 — every economic action including reversals carries provenance and is audited there. |
| ⚪ **Inactive (N/A)** | No rollbacks or reversals possible — economic actions are append-only with no undo. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
