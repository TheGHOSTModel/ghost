# T04 — Trades not replayable

> Trades must not be replayable

## Why this matters

Idempotency tokens on every trade submission. Same token must produce exactly one outcome; multiple outcomes for one token is the refund-and-claim primitive.

## Telemetry

**Event:** `trade_submission`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `trade_id` | string | yes | identifier |
| `idempotency_token` | string | yes | value |
| `parties` | array | yes | list of identifiers |
| `outcome` | string | yes | value |

**Example payload:**

```json
{
  "event": "trade_submission",
  "trade_id": "<trade_id>",
  "idempotency_token": "<idempotency_token>",
  "parties": [
    "<id_1>",
    "<id_2>"
  ],
  "outcome": "<outcome>"
}
```

## Detection signal

```
Same idempotency_token producing more than one outcome
```

## Threats this catches

- Trade replay
- Refund-and-claim exploit
- Token reuse

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Trade submissions carry an idempotency_token and repeated tokens with differing outcomes alert. |
| 🟡 **Latent** | Idempotency tokens are accepted but not enforced for single-outcome semantics. |
| 🔵 **Collapsed** | Covered by G08 — action replay protection already covers trade replay. |
| ⚪ **Inactive (N/A)** | No trade or transactional action exists — see T01. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
