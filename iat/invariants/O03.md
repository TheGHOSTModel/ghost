# O03 — Resist traffic floods

> Services must resist traffic floods and request abuse

## Why this matters

Per-source rate limits on every external endpoint. The signal is both throttling decisions and sustained near-limit traffic.

## Telemetry

**Event:** `rate_limit_event`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `source_id` | string | yes | identifier |
| `endpoint` | string | yes | value |
| `decision` | string | yes | decision/outcome enum |
| `current_rate` | number | yes | numeric value |
| `limit` | number | yes | numeric value |

**Example payload:**

```json
{
  "event": "rate_limit_event",
  "source_id": "<source_id>",
  "endpoint": "<endpoint>",
  "decision": "allow",
  "current_rate": 0.87,
  "limit": 0.87
}
```

## Detection signal

```
decision == 'throttled' OR current_rate near limit sustained
```

## Threats this catches

- Volumetric DoS
- Slow-loris
- Request-spam abuse

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Per-source rate limits enforced on every endpoint, with decisions emitted and sustained near-limit traffic alerted. |
| 🟡 **Latent** | Rate limits configured but decisions are not emitted as events. |
| 🔵 **Collapsed** | Covered by O07 (operational limits) — rate limits are one class of the same limit-enforcement event. |
| ⚪ **Inactive (N/A)** | No external request surface — the system has no inbound network endpoints. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
