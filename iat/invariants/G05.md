# G05 — Timing not exploitable

> Timing must not be exploitable for competitive advantage

## Why this matters

Latency and clock drift must not be weaponizable. Action timing outside tolerance, or bursts inside a server race window, indicate lag-switching or time-warp abuse.

## Telemetry

**Event:** `action_timing`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `player_id` | string | yes | identifier |
| `action_id` | string | yes | identifier |
| `server_ts` | string | yes | ISO-8601 timestamp |
| `client_ts` | string | yes | ISO-8601 timestamp |
| `drift_ms` | integer | yes | numeric value |

**Example payload:**

```json
{
  "event": "action_timing",
  "player_id": "<player_id>",
  "action_id": "<action_id>",
  "server_ts": "2026-05-27T12:34:56Z",
  "client_ts": "2026-05-27T12:34:56Z",
  "drift_ms": 42
}
```

## Detection signal

```
drift_ms outside tolerance OR action burst within race window
```

## Threats this catches

- Lag-switching
- Race-condition exploit
- Time-warp

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Action timing (client vs server timestamps, drift, burst patterns) is sampled and compared against tolerance bounds. |
| 🟡 **Latent** | Timestamps are captured on both sides but no drift analysis runs. |
| 🔵 **Collapsed** | Folded into G02 (input rate) — timing exploits manifest as rate anomalies and are caught there. |
| ⚪ **Inactive (N/A)** | No timing-sensitive actions exist — turn-based or untimed interaction model with no competitive timing. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
