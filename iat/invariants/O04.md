# O04 — Failures non-cascading

> Failures must be isolated and non-cascading

## Why this matters

Circuit breakers between services. Open breakers, or correlated upstream failures, indicate cascade conditions in progress.

## Telemetry

**Event:** `circuit_breaker_state`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `service` | string | yes | value |
| `state` | string | yes | decision/outcome enum |
| `last_open_ts` | string | yes | ISO-8601 timestamp |
| `downstream_failures` | number | yes | numeric value |

**Example payload:**

```json
{
  "event": "circuit_breaker_state",
  "service": "<service>",
  "state": "<state>",
  "last_open_ts": "2026-05-27T12:34:56Z",
  "downstream_failures": 0.87
}
```

## Detection signal

```
Breaker open OR upstream failures correlated across services
```

## Threats this catches

- Cascade failure
- Thundering herd
- Cross-service contagion

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Circuit breakers are deployed between services and state transitions emit events with downstream failure counts. |
| 🟡 **Latent** | Breakers exist but transitions are not emitted as queryable events. |
| 🔵 **Collapsed** | Covered by O10 (service health) — breaker state is one of the SLI dimensions emitted there. |
| ⚪ **Inactive (N/A)** | No multi-service architecture — the system is a single process with no inter-service calls. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
