# S07 — Hierarchies resist abuse

> Social hierarchies must resist abuse

## Why this matters

Role-scoped actions must check a hierarchy rule_id. Missing rule_id, or a role exceeding its declared permissions, is mod abuse or role escalation.

## Telemetry

**Event:** `role_action`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `actor_role` | string | yes | value |
| `target` | string | yes | value |
| `action` | string | yes | value |
| `hierarchy_rule_id` | string | yes | identifier |
| `decision` | string | yes | decision/outcome enum |

**Example payload:**

```json
{
  "event": "role_action",
  "actor_role": "<actor_role>",
  "target": "<target>",
  "action": "<action>",
  "hierarchy_rule_id": "<hierarchy_rule_id>",
  "decision": "allow"
}
```

## Detection signal

```
Role exceeding hierarchy permissions OR rule_id missing
```

## Threats this catches

- Role escalation
- Mod abuse
- Privilege chaining

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Role-scoped actions check a hierarchy rule_id and emit the actor role, action, and decision. |
| 🟡 **Latent** | Roles exist but hierarchy rules are not yet evaluated or emitted. |
| 🔵 **Collapsed** | Covered by S09 — every community action emits actor context that includes role. |
| ⚪ **Inactive (N/A)** | No role hierarchy — the application has a flat user model. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
