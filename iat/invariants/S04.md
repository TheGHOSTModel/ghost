# S04 — Coordination observable

> Coordination patterns must be observable

## Why this matters

Coordination — across accounts, IPs, devices — must be sampled and scored. The signal catches brigading and botnet behavior that single-account heuristics miss.

## Telemetry

**Event:** `coordination_signal`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `account_set` | array | yes | list of identifiers |
| `signal_type` | string | yes | value |
| `strength` | number | yes | numeric value |
| `window` | integer | yes | numeric value |

**Example payload:**

```json
{
  "event": "coordination_signal",
  "account_set": [
    "<id_1>",
    "<id_2>"
  ],
  "signal_type": "<signal_type>",
  "strength": 0.87,
  "window": 42
}
```

## Detection signal

```
Strong correlation across accounts without organic explanation
```

## Threats this catches

- Coordinated harassment
- Botnet activity
- Brigading

## Status guidance

The status field above is **blank by default** — the framework is application-agnostic. Set it for your application using the criteria below.

| Status | When to mark it |
|--------|-----------------|
| 🟢 **Active** | Cross-account coordination is sampled and scored, and strong correlations without organic explanation alert. |
| 🟡 **Latent** | Cross-account signals are collected but not yet correlated or scored. |
| 🔵 **Collapsed** | Covered by S08 — pair-level interaction signals capture the relevant coordination. |
| ⚪ **Inactive (N/A)** | The application has no multi-account interaction or no notion of distinct accounts. |

When marking **Latent**, **Collapsed**, or **Inactive (N/A)**, also fill in the `rationale` field in the frontmatter with a one-line justification.
